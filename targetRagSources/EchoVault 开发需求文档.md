# **EchoVault：轻量级隐私个人知识库开发需求文档**

## **1\. 项目概述**

### **1.1 项目目标**

打造一个极致轻量、本地优先（Local-First）的 RAG（检索增强生成）中间件。通过本地 NPU 加速的小参数模型（SLM），实现对个人隐私数据的预处理、语义压缩与安全检索，为云端或本地大模型提供高质量、低成本的上下文增强。

### **1.2 核心价值**

* **极致隐私**：所有敏感信息脱敏与向量化均在本地断网环境完成。  
* **极致轻量**：安装包 \< 15MB，运行内存控制在 1GB 左右。  
* **无感体验**：利用 NPU 实现瞬时索引与上下文压缩。

## **2\. 技术架构 (Technical Stack)**

* **应用框架**：Wails (Go \+ System WebView) —— 追求最小二进制体积与原生性能。  
* **后端语言**：Go 1.22+ (利用 Goroutine 处理并发索引流水线)。  
* **向量数据库**：SQLite-vec (C 扩展) —— 单文件存储，SQL 与向量检索一体化。  
* **推理引擎**：ONNX Runtime / llama.cpp (CGO 绑定) —— 支持本地 NPU 加速。  
* **本地模型 (SLM)**：  
  * **Embedding**: BGE-Micro-v2 (约 20MB, 384维)。  
  * **Distillation/Privacy**: Qwen3-0.5B 或 Phi-3-mini-0.5B (4-bit 量化)。

## **3\. 核心功能模块设计**

### **3.1 异步无感多格式索引流水线 (Indexing Pipeline)**

系统需支持多种文件格式的深度解析与切片：

1. **文件预处理器 (Format Parsers)**：  
   * **PDF/Word/PPT**：使用 `MuPDF` 或 `unioffice` 提取文本。优先转译为带层级的 Markdown。  
   * **Excel/CSV**：按行处理，每行数据自动关联表头信息，确保行列语义不丢失。  
   * **图片**：集成 `RapidOCR (ONNX)`。在本地 NPU 上运行 OCR，将文字提取结果作为图片向量化的输入。  
   * **代码**：集成 `Tree-sitter`。基于 AST 识别函数体、类定义，按逻辑块切片而非纯字符数切片。  
2. **智能语义切片 (Semantic Chunking)**：  
   * **策略**：优先按 Markdown 标题层级切分，其次是段落，最后是语义符号。  
   * **参数设定**：`ChunkSize: 512 tokens`, `Overlap: 10%`。  
3. **向量化 (Embedding)**：  
   * 调用本地 BGE 模型。支持 NPU 加速以降低 CPU 负载。

### **3.2 隐私扫描中间件 (Privacy Guard)**

在数据向量化及离站前执行：

* **第一层：正则过滤**。匹配 API Keys、密匙（RSA/SSH）、数据库连接串、PII（手机号、邮箱）。  
* **第二层：SLM 语义识别**。利用 0.5B 模型识别文本中模糊的人名、项目名。  
* **脱敏策略**：提供 `[MASK]` 遮盖或 `[ENTITY_TYPE]` 占位符替换。

### **3.3 语义压缩与内容蒸馏 (Context Distillation)**

* **原理**：利用 SLM 将召回的 Top-K 原始片段重写为精炼的事实列表。  
* **目标**：将 Context 长度压缩至原来的 20%-30%，大幅降低云端 Token 消耗。

### **3.4 存储管理 (Storage)**

* 使用 `sqlite-vec` 存储。  
* **表结构设计**：  
  * `knowledge_base`: 存储文本原文、文件类型、元数据（路径、页码/行号、修改时间）。  
  * `vec_idx (Virtual Table)`：存储对应的向量数据。

## **4\. 资源占用指标 (Resource Requirements)**

| 指标 | 目标值 | 备注 |
| :---- | :---- | :---- |
| **磁盘空间** | \< 1GB | 含模型文件（SLM \+ Embedding \+ OCR-Lite） |
| **静态内存** | \< 150MB | 仅 UI 与监听服务运行 |
| **推理内存** | 800MB \- 1.2GB | 4-bit 模型推理峰值 |
| **OCR 响应** | \< 200ms/张 | 基于本地 NPU 加速 |

## **5\. 交互需求 (UI/UX)**

* **全局浮窗模式**：支持类似 Raycast 的快捷键唤起。  
* **模型/引擎管理**：用户可自定义 OCR 引擎开关及模型切换。  
* **隐私实时反馈**：在索引或输出时，通过微小动效提醒用户哪些内容已被本地脱敏。  
* **结果回溯**：点击搜索结果，可直接定位并打开原始文件（支持定位到 PDF 对应页或代码行）。

## **6\. 开发优先级 (Roadmap)**

1. **Phase 1 (基础版)**: Wails 框架与 `sqlite-vec` 集成，实现 TXT/MD 索引。  
2. **Phase 2 (多格式增强)**: 集成 PDF/Office 解析库与 Tree-sitter 代码解析。  
3. **Phase 3 (多模态扩展)**: 集成 ONNX 版 OCR 引擎，实现图片内容检索。  
4. **Phase 4 (智能压缩)**: 完善隐私正则集与 0.5B 模型脱敏/压缩逻辑。  
5. **Phase 5 (企业级演进)**: 开启分布式架构与企业数据湖集成测试。

## **7\. 架构演进：企业级商用扩展 (Future Direction)**

针对商用场景，EchoVault 将向高性能分布式知识基础设施演进：

### **7.1 分布式高可用架构**

* **多节点部署**：支持分片（Sharding）存储，利用负载均衡（Load Balancing）分摊大规模并发检索压力。  
* **容灾备份**：引入 Raft 协议保证元数据一致性，支持多地数据备份与 S3 兼容的对象存储冷备。

### **2.2 企业级权限与隐私控制**

* **组织架构集成**：支持部门间、集团间多层级的知识共享权限（Vector-Level RBAC）。  
* **多级脱敏**：针对不同密级（公开、内部、机密）自动匹配脱敏强度。

### **2.3 企业数据湖集成 (Data Lake Integration)**

* **全域提炼**：打通企业内文件服务器、代码仓库（Git）、内部数据库（DB），实现全格式实时入湖。  
* **近实时同步**：基于事件驱动架构（如 Kafka/NATS），员工保存文件的瞬间即完成向量提炼，供内部大模型调用。

## **8\. 安全性声明**

* **离线原则**：本地版本索引、OCR 与压缩逻辑严禁发起外部网络请求。  
* **数据加密**：本地 `.db` 文件可选支持 SQLCipher 加密存储。  
* **企业版安全**：商用环境支持联邦检索，确保“数据不动，知识流动”。
